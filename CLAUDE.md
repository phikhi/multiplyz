# CLAUDE.md — multiplyz

## Projet
Jeu web de maths **ludique** pour aider une fille de **8 ans (CE1→CE2)** à combler ses lacunes : **compléments à 10, addition, soustraction, multiplication**. Jeu **sans fin**, collection de créatures, mascotte **Teddy** (son doudou Steiff). **App familiale perso** (pas un produit public).

## État
**Greenfield : specs complètes, code pas encore scaffoldé.** → **Lire la spec concernée avant d'implémenter une feature.**

## Stack (détail : `STACK.md`)
Next.js (App Router, **runtime Node**, pas edge) + React + TS + **Tailwind v4** + `tokens.css`. **SQLite local** (`better-sqlite3`, WAL) + **Drizzle**. Zustand. **PWA online-first**. Hébergement **VPS OVH via Laravel Forge** (daemon Node `next start` + Nginx + Let's Encrypt). Génération d'assets IA via **worker daemon**.

## Règles non négociables (en codant)
- **Tokens** : jamais de valeur en dur (couleur/espacement/typo/rayon) → `var(--…)` ou alias Tailwind. Source = `tokens.css`.
- **Copy** : français, **tutoiement**, posture croissance (« pas encore », jamais « faux »), voix de Teddy. **Strings centralisées** (zéro texte en dur).
- **Pédagogie** : **no-fail**, temps mesuré en silence, maîtrise = **juste + rapide**. La logique maîtrise/sélection vit **côté serveur** → suivre `ENGINE.md`, ne pas réinventer.
- **Serveur/données** : serveur = **source de vérité**, **online-first**, **runtime Node**, écritures **idempotentes**, progression **monotone**. SQLite **WAL**.
- **Migrations (Drizzle/SQLite)** : jamais `ADD col NOT NULL` sans default (crash sur table **peuplée** ; CI migre une base vide → ne l'attrape pas). Colonne dérivée **calculée par l'app** (non exprimable en SQL, ex. `nameKey` NFC+locale) = **nullable + backfill applicatif** dans `runMigrations` (idempotent `WHERE col IS NULL` + `db.transaction()` atomique + **anti-collision testée par source** — intra-batch ET valeur déjà persistée). **Jamais `.notNull()` de type sur une colonne physiquement nullable** (+ snapshot désync) = **drift snapshot↔SQL proscrit** (rétro #91/#105) : cohérence **schema.ts↔snapshot↔SQL↔base** (`db:generate` **no-op**) ou **table rebuild** ; garde observable `PRAGMA table_info` sur la nullabilité. Éditer une migration **livrée** seulement si **aucune base non-réparable ne l'a jouée** (skip drizzle = **timestamp** → migration éditée **inerte** sur base déjà migrée). Contrat data modifié → **ADR**.
- **Économie** : jamais d'argent réel, **ne bloque jamais l'apprentissage**, doublon → éclats, dépenses **online only**.
- **Sécurité** : PIN **hashé** (argon2/bcrypt), **rate-limit**, single-tenant famille. Jamais de PIN en clair / côté client. **Écriture NO-PIN / bas-privilège réutilisant une fonction d'écriture LARGE (rétro 8.6, #282)** : une server action **no-PIN** (ex. quick-mute enfant écrivant la ligne `household_settings` que le parent possède) qui réutilise une fonction d'écriture **arbitraire/parentale** (`writeHouseholdSettings`) doit prendre un **paramètre scalaire isolé** (`enabled: boolean`) et construire un **patch LITTÉRAL à N champs FIXES dans le corps de l'action** (`writeHouseholdSettings(db, { soundEnabled: enabled === true })`) — **JAMAIS** un objet fourni par le client (le server action reçoit l'input client **brut** au runtime, les types TS ne protègent pas : `{ soundEnabled, volume:999, pin }` posté comme argument doit voir ses clés smugglées **jetées**, jamais spread dans la fonction large). Gardé par la **session** (child-session exigée même sans PIN) + coercition stricte (`=== true`). **Mutation-prouvé par un test deep-equal EXACT** (`toHaveBeenCalledWith(db, { soundEnabled: true })`, PAS `objectContaining`) qui rougit à tout élargissement du patch. **Security review = dimension distincte au panel dès qu'une surface enfant/bas-privilège écrit une donnée partagée.**
- **Assets IA** : prompt de base **verrouillé** (ART), un monde ne pose que `--world-accent`, **Teddy depuis photos réelles**, QA kid-safe + fallback.
- **A11y** : feedback **doublé d'icône** (daltonisme), cibles ≥ 44 px, `prefers-reduced-motion`. Couleur de statut = token texte **constant** (`--color-on-warning`…), jamais un token qui s'inverse par thème (contraste). **Scaffolds visuels** : `--color-text-inverse` réservé à un fond **accent plein** — jamais posé sur un fond **neutre** (piège récidivant #94→#102, rétro #104) ; tout glyphe/chiffre/trait de scaffold visible exige **≥1 test de contraste WCAG à effet observable** (résolution `tokens.css`, `var()` résolus, ratio ≥4.5:1 — pas seulement le nom du token). Pour ces scaffolds, une garde visuelle doit asserter la **géométrie rendue** et/ou la **couleur résolue**, jamais seulement la forme d'un path SVG ou des `data-*` (100% coverage vacuous sur ces derniers, rétro #104). Le test de contraste s'applique **par glyphe/trait rendu distinct**, pas une fois par famille de tokens (deux glyphes différents — étoile pleine/vide, actif/inactif — sur un même scaffold = **deux tests**) ; le **fond de référence** = le fond DOM réellement empilé derrière ce glyphe (un frère empilé n'a pas le fond du médaillon voisin — rétro #125). **Quand une story TOUCHE un écran, auditer TOUS les glyphes rendus qu'il porte** pour la règle de contraste, pas seulement les nouveaux — un glyphe hérité d'une story antérieure reste **livré** par cette PR (rétro #126 : étoiles de #64 sur l'écran de résultats de 5.5, décoratives `--color-star` sur fond neutre ~1.2:1). **Token nommé du wireframe** : un token déclaré dans `tokens.css` **sans consommateur DOM** signale un **élément manquant**, pas un élément prêt — quand un wireframe **nomme** une métaphore structurante (« chemin », « piste », « fil »), poser une garde à effet observable que le DOM **rend l'élément nommé et consomme son token** (déclarer le token ne prouve pas le rendu ; piège invisible aux tests unitaires et à la capture, rétro #125). Cette règle « **déclaré ≠ rendu/consommé** » s'étend au-delà des tokens : (a) tout **param ⚙️ nommé par une spec** comme comportement calibrable (longueur du boss, seuil, palier) — le déclarer en config + le couvrir par un test de **présence** ne prouve pas qu'il **agit** ; exiger un test à effet observable qui **exerce le vrai chemin runtime** et rougit si le comportement retombe au défaut (rétro #127 : `bossQuestionCount` inerte, boss joué comme un niveau normal de 10 Q derrière un panel gates/coverage/mutation vert — récidive fidélité-modèle #95/#123, vue par game-design, pas par l'AC-littérale PO) ; (b) toute **valeur de layout structurante chiffrée** nommée par un wireframe (nombre de colonnes, gabarit) — `auto-fill`/`auto-fit` n'honore pas un compte fixe ; garder par un test non-vacuous du **compte rendu** (rétro #127 : « 3 colonnes » WIREFRAMES §8 rendues 1-2 sur téléphone). **Rendu + token consommé + contraste résolu vert ≠ PIXEL VISIBLE (rétro #170)** : un élément peut être présent au DOM, consommer son token, passer sa garde de contraste ≥X:1 résolue **et rester invisible à l'écran** — recouvert par un frère opaque (**occlusion / z-index**), `height:0`, hors cadre, clippé, mauvaise géométrie `top/height`. **jsdom ne fait AUCUN layout** → aucun test unitaire n'attrape l'occlusion (#170 : trait `--map-node-path-color` ≥3:1 mais peint DERRIÈRE le médaillon `zIndex:1` → invisible ; 4/4 reviewers ont validé le token, aucun les pixels ; corrigé en déplaçant le SVG dans la gouttière `top:var(--map-node-size)`). Donc pour tout élément **superposé/positionné** (SVG, overlay, badge, `absolute`), la garde de contraste/rendu doit être **doublée** (a) d'une **capture Playwright dont les PIXELS sont réellement analysés** (ouvrir l'image, regarder — pas seulement la générer) ET/OU (b) d'une **garde E2E sur la géométrie rendue réelle** (`boundingClientRect` : l'élément n'est pas recouvert, sommet ≥ bas du frère opaque). « Token vert » prouve la mécanique, **jamais** la visibilité. **La géométrie RAISONNÉE ne prouve pas la non-occlusion non plus (rétro #190, extension #170)** : pour tout élément superposé/positionné, une **marge calculée « sûre »** (« l'avatar s'étend de −40 à +8px, gap 32px → aucune occlusion ») peut être **empiriquement fausse** — en #190 l'avatar Teddy de 48px débordait **réellement de 6px** sur le médaillon du nœud **amont** (gouttière 32px), et **5 reviewers + l'orchestrateur + le pixel-look sur capture fixture** l'ont tous déclaré sûr ; seul un **assert `boundingClientRect` en vrai navigateur** l'a attrapé. Donc : (a) ne jamais substituer un raisonnement géométrique (même convergent entre reviewers) à un assert de layout réel ; (b) garder la non-occlusion contre **TOUS les frères empilables** — l'élément « sous » l'overlay ET les frères **adjacents/amont** dans la gouttière (pas seulement l'élément courant). **`opacity`/`filter`/fond semi-transparent d'un ancêtre casse le contraste résolu (rétro #226, 3ᵉ axe du piège « résolu ≠ peint » après occlusion #170 et token-non-résolu #104)** : un token de couleur **résolu ≥4.5:1 ne prouve PAS** le contraste dès qu'un `opacity`/`filter`/fond semi-transparent d'un **ancêtre** s'empile — `opacity` CSS composite **tout le sous-arbre** (texte inclus) vers le fond, le pixel réellement peint est la couleur **post-blend** (ex. #226 : owner-« Supprimer » désactivé sous `opacity:0.55` = **2.26:1 peint** alors que le test « résolu » le déclarait ≥4.5:1 sur les tokens pleine-opacité = **test vacuous**, capté par le seul Frontend recalculant le WCAG peint à la main ; 4 reviewers ingénierie + pixel-look du build l'ont manqué). Donc tout test de contraste sur un élément susceptible d'être **sous opacité/blend** (désactivé, pending, fondu, overlay) doit résoudre l'**état COMPOSITÉ réellement peint** (lire l'`opacity` rendue + blend `mixSrgb` sur le fond DOM réel), jamais seulement la paire de tokens de base ; **corollaire de conception** : ne pas diluer du **texte** par une `opacity` de sous-arbre — garder le texte **plein-alpha** (token déjà ≥4.5:1) et signaler l'état par `cursor:not-allowed`/`aria-disabled`/bordure atténuée. **Axe granularité de la garde (rétro #260, extension #226)** : une garde anti-#226 par **valeur d'opacity interdite unique** (`expect(opacity).not.toBe("0.5")`) est **vacuous** — toute autre valeur diluante de la même classe (`0.55`…) la désarme silencieusement ; exiger (a) le rejet de **toute** opacité diluante (`expect(["", "1"]).toContain(opacity)`) et/ou (b) la composition de l'opacity **réellement rendue** dans le calcul (`paintedContrast`/`mixSrgb` post-blend ≥4.5:1), et **muter la garde avec une valeur représentative de la classe**, jamais un cas unique. **Le FOND de la CAPTURE peut MENTIR — 4ᵉ axe « résolu ≠ peint » après occlusion #170, jsdom #104, opacity composité #226 (rétro #203)** : un **pixel-look sur une capture n'est PAS une preuve de contraste** quand le **fond de la fixture diffère du VRAI fond cible** de déploiement — en #203 la bordure `--map-tiles-border` (`--color-border-primary`) a été validée « lit comme décor » par l'**orchestrateur + game-design + PO** sur une capture à fixture **photo texturée rayée** (contraste de luminosité **fortuit** avec le motif), alors qu'elle résolvait à **1.18:1 = fonctionnellement invisible** contre le vrai `--world-bg-tint` tint-seul (état pré-photo owner-gated) → AC « décor » **vacuous** ; seul le Frontend recalculant le **contraste composité RÉSOLU contre le vrai fond** (6 accents curatés × 2 thèmes) l'a attrapé. Donc tout élément décoratif/non-texte dont la cible de contraste est un **fond SPÉCIFIQUE** (tint per-monde, état pré-asset) exige un **test de contraste composité RÉSOLU contre le VRAI fond cible** (jamais la fixture ni le seul pixel-look de capture — la capture peut tromper par texture non représentative) ; réutiliser le harness `resolveTokenColor`/`mixSrgb`/`contrastRatio`/`CURATED_ACCENTS` déjà présent. Symétrie #190/#299 : la seule preuve recevable est le **calcul résolu contre le vrai fond**, pas le regard sur une fixture. **Audit binaire au ref** : utiliser **`git cat-file -p <blob>`**, jamais `git show <ref>:<fichier>` piped — le proxy rtk corrompt la sortie binaire (octets ≥0x80 → U+FFFD) → faux « fichier/PNG corrompu » (rétro #190, #126/#147-class). Cette classe s'étend au **stdout texte** de `prettier`/format (pas seulement au binaire) : le proxy peut renvoyer un succès **canné** pour `prettier --check`/`--version` indépendamment de l'état réel du dépôt (tell : `--version` ne retourne pas un numéro de version) — vérifier tout gate de formatage via le **binaire direct** (`node node_modules/prettier/bin/prettier.cjs --check .`), jamais le stdout rtk-proxied (rétro 8.2). **Cette classe s'étend aux ÉCRITURES git** (`merge`/`pull`/`checkout`) : le proxy peut renvoyer un « Fast-forward » **canné** alors que `HEAD` n'a PAS bougé (rétro 8.3 : `git merge --ff-only` a rapporté « Updating..Fast-forward » succès pendant que `HEAD` restait au commit précédent → le subagent a d'abord édité sur la MAUVAISE base, quasi fausse-régression) — vérifier toute écriture git rtk-proxied par un `git rev-parse HEAD` / `git rev-list --parents` / `ls-remote` **binaire direct**, jamais le stdout du merge/pull. **Cette classe s'étend à la CAPTURE DE PATCH (rétro #299)** : `git diff <fichier> > sortie.patch` sous rtk produit une sortie **prettifiée** (en-tête `--- Changes ---`, indentation à 2 espaces, pied `+N -M`) donc **non applicable** en patch — ne jamais construire un `.patch` via `git diff > fichier` sous rtk ; capturer le contenu via Read/Write direct, ou au minimum **vérifier `git apply --check`** sur le patch produit **AVANT** toute opération destructive (`checkout --`, `reset --hard`) sur la source qu'il est censé sauvegarder (rétro #299 : une modification non committée de `SKILL.md` sauvegardée ainsi puis un `git checkout --` sur le fichier — le patch était corrompu, la modif n'a été récupérée que parce qu'elle vivait encore dans le contexte de conversation). Tell commun à toute la famille : le proxy **reformate** une sortie destinée à être consommée par une machine. **Token CSS dérivé (variante #184 côté CSS, rétro #182)** : un token `color-mix`/`calc` **référençant une autre var** déclaré à `:root` **ne se re-dérive PAS** sous une **surcharge inline descendante** de la var source (la substitution `var()` d'un custom-property se résout au niveau de **déclaration**, pas d'usage) — ex. `--world-bg-tint: color-mix(…var(--world-accent)…)` à `:root` reste **neutre** même quand `<main>` surcharge `--world-accent` inline. « Dérivé automatiquement » est alors un **faux-dérivé dormant** (invisible aux tests unitaires — jsdom ne résout pas `color-mix`) : re-déclarer le token dérivé **au niveau de la surcharge** de sa var source, ou l'exercer par une garde à effet observable (valeur calculée en vrai navigateur). Un commentaire qui affirme la dérivation per-état non atteinte = tell #164 à corriger. **Effet MOUNT-ONLY (focus-au-montage, animation d'entrée, mesure layout) sur un élément de MÊME TYPE échangé entre branches d'état SANS `key` distinct = jamais ré-exécuté (rétro #244, 4ᵉ axe « pipeline vert ≠ correct » après pixels #170, contraste composité #226, i18n #239)** : quand un composant retourne le même élément (ex. `<StatusMessage>`/`<h1 ref={focusOnMount}>`) à la même position depuis plusieurs branches (`if kind===… return`), React réconcilie chaque transition d'état en **UPDATE** (même Fiber, même nœud DOM réutilisé) — un ref-callback `focusOnMount`/`useEffect([])` ne se réinvoque que sur mount/unmount RÉEL, **jamais sur la transition**. Un test jsdom `expect(document.activeElement).toBe(heading)` **passe VACUOUSLY** : le même nœud (focalisé au 1er état) reste `activeElement` pendant que son `textContent` mute en place → assertion déterministe mais qui **ne prouve pas** que l'effet s'est ré-exécuté sur l'état cible (l'écran prioritaire — ex. `locked` — hérite silencieusement du trou ; un SR ne réannonce pas un élément déjà focalisé dont le texte change sans nouvel événement focus). **jsdom-déterministe ≠ mécanisme-prouvé.** Garde : (a) `key` DISTINCT par état sur l'élément échangé → force un vrai remount → l'effet se ré-exécute ; (b) le test exerce la **TRANSITION** (rerender état→état) avec un **`blur()` intercalé** pour distinguer « effet ré-appliqué » de « même nœud resté focalisé » → il doit ROUGIR sans le `key`. Capté par l'instrumentation empirique (spy `.focus()` / log du ref-callback), **jamais** par la couverture ni la mutation de la garde listée ni le pixel-look. Prolonge #36/#397 (« chaque substitution de contenu exige un audit du focus » couvre AUSSI le cas où le contenu se substitue **sans retirer l'élément focalisé du DOM »). **Corollaire drain a11y** : une story de DRAIN a11y auditant un écran doit couvrir la **FAMILLE de défaut ENTIÈRE dans ses fichiers touchés AU BUILD** (grep `opacity:` sur tout élément désactivé/pending/fondu ; #226 récidive dans les mêmes fichiers édités) — #126 est une **obligation de build**, pas un catch de review (rétro #250 : 3 opacity-dilutions #226 restantes dans les fichiers édités pour #222/#244, une visible dans la capture même de la PR). **Corriger une occlusion d'overlay `fixed` en GATANT sa route ne la SUPPRIME pas, elle la RELOCALISE sur le contenu des surfaces nouvellement autorisées (rétro #278, extension #170/#190)** : en #278 le gating anti-onboarding de l'invite PWA a corrigé l'écran signalé MAIS l'overlay `fixed;top` recouvrait alors le `<h1>` PROPRE des surfaces cible (« Ma collection 🐾 » de `/collection` réduit à un fragment d'emoji ; idem `/carte`) — capté seulement au 3ᵉ round par le pixel-look PO, manqué par mon propre pixel-look round 2 qui n'avait vérifié que l'overlay + l'écran initial. Donc : (a) le pixel-look d'un élément **superposé/positionné** doit auditer le **contenu HÔTE réel** (son `<h1>`/titre focus-managé) de **CHAQUE route** où il apparaît, pas seulement l'élément posé ni l'écran du bug initial (extension #170/#126 « auditer TOUS les glyphes rendus » au contenu recouvert) ; (b) **préférer la non-occlusion STRUCTURELLE** — un élément **EN FLUX** (premier enfant, `position` absente) qui **réserve un espace réel** et pousse le contenu ne peut PAS occulter par construction et ne peut PAS relocaliser le bug — à la non-occlusion **GARDÉE** (overlay `fixed` + garde `boundingClientRect`, correcte mais qui peut manquer une surface non testée) ; (c) la garde E2E de non-occlusion doit asserter la **coexistence overlay+titre hôte** (`elementFromPoint` sur `main h1` ET `overlayBottom <= titleTop`), jamais seulement la non-occlusion de l'overlay lui-même. **Corollaire test React 19** : asserter l'ABSENCE de `console.error` pour prouver un garde anti-setState-post-unmount est **vacuous** (React 19 a supprimé le warning « unmounted component » → l'assertion ne peut plus rougir, #143) — observer directement le mécanisme (spy `add`/`removeEventListener`, spy `.focus()`), jamais un warning de framework supprimé.
- **Paramètres `⚙️`** (notés dans les specs) = à **calibrer**, centralisés dans un fichier de config, jamais figés en dur sans raison.
- **Tests/CI** : seuils coverage **armés** (`thresholds` + `all:true`) — un `exclude` ne doit **jamais vider le scope** (100% vacuous = gate creux) ; exclude réservé au boilerplate framework non testable. Toute **garde / branche de repli** (anti-adjacence, anti-doublon, anti-collision, borne exacte) exige **≥1 test à effet observable** — un test qui **échoue si la garde est retirée/mutée**, pas seulement qui l'exécute ; le 100 % de coverage (même non-vacuous, `all:true`) ne suffit pas à lui seul (rétro #60/#61, PR #77). **Structure procédurale/dérivée** (composée à la volée) dont les **positions servent de clé persistée** (ex. `progress.level_index` vs géométrie de carte) ⇒ la **géométrie (nombre d'éléments + positions) doit être INVARIANTE à l'état runtime** (ex. la dette ne change pas le nombre de nœuds, seul un attribut non-clé comme le `type` peut varier) — garde = **test d'invariance de géométrie à effet observable** (même sortie structurelle quel que soit l'état dynamique ; casse si un élément est ajouté/retiré dynamiquement — rétro #123). **Test de rollback/atomicité** : la panne induite doit frapper **l'opération protégée par la transaction** (2ᵉ écriture), **jamais** un statement **en amont** (lecture/garde de pré-condition) qui court-circuiterait avant que la 1ʳᵉ écriture ne soit tentée — sinon le test reste **vert sans la transaction** (vacuous, 100 % coverage trompeur). `DROP TABLE` d'une table **lue en amont** = **anti-pattern** ; casser plutôt l'**écriture gardée** (colonne/contrainte manquante) pour que la panne survienne *après* la 1ʳᵉ écriture. Preuve exigée = retirer `db.transaction` **casse précisément ce test** (rétro #122). Cette exigence vise les transactions **multi-écritures** (≥2 écritures — faire échouer la 2ᵉ *après* la 1ʳᵉ pour observer le rollback). Une transaction sur **écriture unique** n'a **aucun état partiel à annuler** ⇒ **aucun test de rollback non-vacuous n'existe** (il passerait avec **ET** sans transaction) ⇒ **documenter honnêtement la garde** (ne jamais sur-revendiquer un anti-TOCTOU/atomicité *testé* qu'aucun test ne prouve), **PAS** ajouter un test qui resterait vert sans la transaction (lui-même vacuous, donc interdit — rétro #124). Ce principe s'étend à **toute branche/garde sans effet observable distinct de son absence** (pas seulement les transactions à écriture unique) : un prédicat/condition **redondant avec la sémantique du moteur** (ex. `WHERE col IS NOT NULL` sur un index UNIQUE composite, redondant car SQLite traite déjà les NULL comme distincts) **n'est pas une garde testable** — le **SUPPRIMER (simplifier)**, jamais garder la branche en revendiquant une couverture ni ajouter un test qui resterait vert sans elle. « **Correct** » (sémantiquement juste) ≠ « **testable** » ≠ « **nécessaire** » (rétro #143). **Tell commentaire↔code (récurrent #162→#164)** : tout **commentaire / titre de test / corps de PR / message de commit** qui affirme un **mécanisme précis** (nom d'API, garde, seuil, invariant — ex. « idempotence via `onConflictDoNothing` » alors que le code fait `onConflictDoUpdate`) se vérifie en review : le code annoté doit le réaliser **littéralement**. Écart = **defect tell** à corriger **même sans bug fonctionnel** derrière (drift intention↔implémentation = piège de maintenance) — corriger le libellé (ou le code s'il est faux), jamais laisser l'affirmation mentir ; vaut pour le **reçu** autant que le code (prolonge #127/#143/#155 côté documentation). **Le BRIEF de délégation de l'orchestrateur est aussi une source de citations (rétro #259, extension #164)** : une **réf `spec:ligne` fausse** dans le brief (ou héritée telle quelle d'une issue/`needs-owner`) se **recopie en aval** dans le code + l'**ADR** + les **specs canoniques** (plus dur à défaire qu'un commentaire) et n'est captée que par la review **FIDÉLITÉ** (ex. #259 : `PRODUCT :108` = titre §3.6, alors que la règle démote vit en `ENGINE §2` — recopié dans 6 fichiers committés, vu par game-design seul, pas par les 4 reviewers ingénierie). **Vérifier toute citation `spec:ligne` du brief AVANT de l'embarquer** (grep la ligne : dit-elle littéralement ce qu'on lui attribue ?), jamais recopier une réf héritée sans la re-vérifier. **Deux axes de récidive #259/#164 (rétro 8.3, PR #276)** : (a) un **balayage de citation/token** (ex. « DETAILS §22 → §3 ») doit grep le **dépôt ENTIER** (config, `.env.example`, docs), **jamais `src/` seul** — un sibling non-src qui traîne rend le reçu « citations corrigées » **littéralement faux** #164 (capté par la seule review FIDÉLITÉ, invisible aux greps `src/`) ; (b) dans un doc canonique (ADR/spec), **ne jamais citer `spec:l.N` quand la MÊME édition décale les lignes** — une insertion d'1 ligne dans `DETAILS §3` a périmé mes propres réfs `§7 l.78` de l'ADR fraîchement écrit → citer **section + ligne NOMMÉE** (« §7 ligne Réglages », insensible au décalage). Ne pas rationaliser un `§N` qui grep-e à rien comme « réf de ligne cohérente » (fait à tort en #276) : disambiguïser ou corriger. **Un GATE de complétude de balayage (grep) est lui-même une garde à effet observable #60 — FORME-complet + TYPE-complet + validé ADVERSARIALEMENT avant que sa sortie vide ne prouve l'exhaustivité (rétro #296, extension #259/#164/#60)** : quand le LIVRABLE d'une story EST l'exhaustivité d'un balayage (citations, tokens), le gate doit matcher (a) toutes les FORMES (composée `§3/§25-32` après un `/§`, lien markdown `[X.md](…) §N`, suffixe-ligne `§3 :29` — pas seulement le token immédiat `X §N`) ET (b) tous les TYPES de fichiers trackés (exclure SEULEMENT le vendored `node_modules`/`coverage`/`.next` — JAMAIS un filtre `*.ts/*.tsx/*.md` qui masque `.env.example`/config, récidive #259 axe-a). Un « gate VIDE (0 match) » non-adversarialement-validé = reçu #164 : avant de le croire, prouver que le gate ROUGIT sur un survivant CONNU (pré-fix ou planté). En #296 le balayage AC3 a pris 3 rounds — gate forme-incomplet (round 1→2) puis type-filtré (round 2→3, le brief de fix de l'orchestrateur AYANT perpétué le filtre) ; chaque miss laissait « gate VIDE » littéralement faux avec des survivants vivants, capté SEULEMENT par la review FIDÉLITÉ (game-design + reviewer de la lane du type de fichier), invisible au gate incomplet que l'orchestrateur ET QA utilisaient. **Compte agrégé de « N tests rouges » dans un reçu = tell fragile (rétro #173, extension #143/#155)** : revendiquer une mutation-preuve par un **cardinal** (« retrait de la règle X → 6 tests rouges ») est un tell qui casse au moindre test ajouté partageant le seam muté (un nouveau test consommant le même `failInspector`/fixture fait passer 6→7 sans que le code soit faux — récidive 6.5 : reçu 9→3 puis 6→7). Revendiquer la mutation-preuve par **test(s) NOMMÉ(s) qui rougit(ssent)** (« le test `budgets DISTINCTS` rougit si les compteurs re-fusionnent »), jamais par un total agrégé — sinon l'exactitude #143/#155 impose un recompte manuel à chaque test ajouté sur le même seam (coûteux + oublié). Vaut pour le reçu du build ET les titres/commentaires de tests. **Deux gardes qui se recouvrent en outcome ⇒ créditer chacune par SA mutation, jamais par le test « principal » (rétro #206, extension #143/#173)** : quand un **SCOPE/narrowing** (prédicat qui restreint les candidats — ex. `WHERE status='failed'`) et une **garde de correction** (appliquée aux candidats restants — ex. `hasNonFailedJobForWorld`) se recouvrent, un test peut **nommer/créditer** la garde de correction tout en n'étant réellement épinglé que par le SCOPE → **muter la garde de correction le laisse VERT** (le SCOPE écarte l'entrée avant qu'elle n'atteigne la garde). Le crédit « mutation-prouvé » d'une garde de correction n'est valide **que** pour le(s) test(s) qui rougissent sous **SA** mutation, sur une **entrée que le SCOPE ne couvre pas déjà** (#206 : un index à la fois `failed` ET `done`). Vérifier quelle garde épingle chaque test en **mutant CHAQUE garde indépendamment** ; ne jamais supposer que le test « principal » prouve la garde de correction (Security+Backend l'ont approuvé, seule la re-mutation par-garde de QA l'a démasqué). Corollaire opérationnel : GitHub `Closes #a #b #c` n'auto-clôt **que le premier** issue (les `#n` nus suivants ne sont pas parsés) → clôturer chaque sous-item à la main (ou `closes #a, closes #b`). **Reçu de mutation-preuve nommant l'assertion/ligne où le test rougit = vérifié en exécution PLEIN-FICHIER, pas isolé (rétro #186, extension #164/#173)** : un `spy.mockRestore()` **non gardé** (sans `try/finally`) posé après une assertion qui peut lever **fuit le spy dans le test suivant** et **DÉPLACE l'échec observé** (le downstream rougit ailleurs — ex. `undefined`/`toHaveLength` — pas à l'assertion revendiquée), rendant le reçu **littéralement faux** même si la régression reste attrapée (pas de faux-négatif CI, mais tell #164 + valeur de debug dégradée). Garder chaque `spyOn` en `try { … expect … } finally { spy.mockRestore() }` (ou `restoreMocks:true` global) pour que l'échec de mutation atterrisse à l'**assertion nommée**. **Chemin format-RÉEL non prouvé par fixtures null/placeholder (rétro #189, extension #180/#184)** : un chemin de rendu/câblage prouvé **uniquement** avec des fixtures `null`/placeholder laisse le **chemin format-réel non prouvé** — il peut rester **dormant** sans qu'aucun test ne rougisse, notamment si un **outil owner-run** (gitignoré, non committé) qui **écrit des données consommées par le code committé diverge du contrat de format** de ce code (divergence **invisible à la CI par construction** : données gitignorées + code committé correct + fixtures placeholder ; défaut #189 = `gen-socle-assets.local.ts` stockait le ref **absolu** `/generated/socle/…` alors que `isRenderableAssetRef` exige le **relatif** `socle/…` → `background:null` → chemin réel dormant, trouvé en **instruisant** le déblocage, pas en review). Exiger (1) un **test d'intégration au format RÉEL valide → sortie non-null** (pas seulement le cas null/placeholder — ET le cas format-fautif → null), et (2) que tout **outil owner-run producteur de données** honore **littéralement** le contrat de format du consommateur committé (mêmes refs/clés que le **défaut committé**, ici `defaultSocleWriteAsset`). **`pnpm format`** relancé après tout **rebase/résolution de conflit** avant push. **CI `cancelled` ≠ hang de code (rétro #230, companion CI de #126)** : un job e2e/CI qui finit **`cancelled`** (≠ `failure`/`timed_out`) à une durée **≈ une étape setup réseau-lourde** (`playwright install --with-deps` apt, `pnpm install`) alors que le job **n'a aucun `timeout-minutes`** = **stall d'infra / run superseded**, PAS un hang de code — même quand le commit ajoute un test qui « ressemble » au coupable (corrélation « nouveau test + rouge » = **faux-tell**, symétrie côté rouge de « vert inattendu → reçu ment » #210). Un hang de code **déterministe reproduit** au re-run ; un stall d'infra **non** → avant de renvoyer un e2e rouge au build, **re-run une fois le MÊME commit** pour désambiguïser infra-flake vs régression (preuve #230 : `cancelled` 15m05s → re-run **success 1m26s**, code identique). Extension CI de #126. **Assertion permissive qui matche la sortie boguée = garde vacuous ; la copie/grammaire (i18n) = 3ᵉ axe de « pipeline vert ≠ correct » (rétro #239, après pixels #170 et contraste composité #226)** : une regex E2E `/^\d+ jours$/` laisse passer « 1 jour**s** » (pluralisation FR fausse : `n<=1`→singulier, 0 ET 1) — lint+typecheck+**100% coverage**+e2e+build TOUS verts sur un texte grammaticalement faux visible sur tout l'écran, capté uniquement par le **pixel-look qui LIT le texte rendu** (« capture générée » ≠ « texte relu »). **Règle** : tout compteur/texte pluralisé affiché doit (a) réutiliser le patron singulier/pluriel **existant** (`count`/`countPlural`, jamais un gabarit `{n}` figé) ; (b) être verrouillé par un test à la **forme EXACTE par n** (fixtures n=0/1/2 rougissant si le pluriel figé revient) **et** une assertion E2E **non-permissive** (comparer au rendu attendu, pas `\d+`). Corollaire a11y (#125 étendu aux strings) : une `aria-label` string **déclarée mais non consommée** (`aria-hidden` sans alt) est un a11y-gap invisible au pixel-look ET aux tests de rendu — asserter que la string a11y est **consommée** (attribut présent sur l'élément). **Un verrou/mécanisme runtime et sa méta-garde d'exhaustivité doivent faillir en sens OPPOSÉS — les confondre est un defect (rétro #299)** : un **verrou/mécanisme runtime** (ex. lock anti-collision) doit faillir en **OUVERT** (état incertain → laisser-passer ; sinon un bug le gèle en deadlock permanent) ; la **garde d'exhaustivité qui vérifie la complétude du câblage de ce mécanisme** (ex. « chaque sortie propre porte son appel de libération ») doit faillir en **FERMÉ** (ancrage non reconnu → orphelin → test rouge ; sinon faux-vert silencieux, #296). En #299 les confondre a produit le defect bloquant du round 4 : l'exemption de synopsis de la garde de balayage était un match de **préfixe**, donc tout ancrage réel non enregistré en bare-command passait vert malgré son absence du câblage — démasqué seulement en injectant une section factice, les 9 tests restant verts jusque-là. **Quand une story livre à la fois une garde runtime et une méta-garde qui prouve la complétude de la première, énoncer explicitement le sens de défaillance de CHACUNE dès la conception** — elles sont structurellement opposées, jamais interchangeables. **Durcissement DUR de #296 : pour tout livrable dont la VALEUR EST l'exhaustivité, le survivant PLANTÉ est la SEULE preuve recevable — un raisonnement de complétude, même convergent entre reviewers, ne compte pas (rétro #299)** : dans cette PR, **chaque** revendication de complétude établie par raisonnement (câblage complet, sortie propre couverte, balayage bidirectionnel, exemption correcte, identifiants uniques) s'est révélée fausse à un round ultérieur, et **chaque fois** seul le plantage empirique d'un survivant l'a démasquée — jamais une relecture, jamais un accord entre reviewers. Avant de créditer un gate/une garde d'exhaustivité VERT(e), y **injecter un survivant CONNU** (pré-fix ou fabriqué) et vérifier qu'il rougit sur SON test nommé ; sans ce coup de sonde, un « tout vert » reste une **revendication non vérifiée** (#164), pas une preuve. Symétrie exacte avec #190 côté géométrie (une marge calculée « sûre » y avait aussi été démentie par un seul assert de layout réel, jamais par un raisonnement, même convergent).

L'agent est **autonome sur le planning** : synchro état → **triage du backlog** (bloquant / prochain-épic / backlog / gate-déploiement) → **découpage de l'épic** en stories → **choix + parallélisation** (surfaces disjointes en //, partagées séquencées) → boucle story-start→PR→reviews→merge→rétro, sans solliciter le proprio. **Escalade UNIQUEMENT sur le drift** (décisions verrouillées PLAN/pédagogie/éco/sécurité/scope). Rapporter les décisions de planning, ne pas attendre d'aval hors drift.
  - **Contexte & quota** (autonomie longue, cf. `orchestrate` §7) : **build délégué à un subagent isolé** par story (thread principal = conclusions only → contexte plat) ; **tiering modèle** (Opus orchestration/cœur · Sonnet reviewers/mécanique · Haiku exploration) ; **jamais couper au milieu d'une story** (frontière propre = après merge+rétro + checkpoint statut) ; **quota bas → pause propre** à la frontière + **reprise auto-programmée** (skill `schedule` : run unique à l'heure de reset → chaîne auto ; en headless, drift → issue `needs-owner`, pas de reprise). **Reçu du subagent de build = VÉRIFIÉ, jamais cru sur parole** (SHA poussé / gates verts / capture) : avant de créditer ou merger, `git log` doit montrer le commit **sur la branche poussée** + `gh pr view --json headRefOid` concorde + **re-run des gates soi-même** ; reçu **invérifiable** (SHA inexistant, HEAD inchangé) = **tâche non faite** (rétro #126). **SHA véridique ≠ revendications de test véridiques** : une revendication « mutation-prouvé / garde à effet observable » pour une garde précise doit être re-vérifiée en **mutant CETTE garde spécifiquement** (le `WHERE`, la 2ᵉ écriture, le token remappé — pas la feature entière) et en observant le rouge, avant d'être créditée ; ne revendiquer « mutation-prouvé » qu'après avoir muté la garde nommée, sinon écrire « testé (exécuté), mutation non vérifiée » (rétro #143 : un prédicat partiel « mutation-prouvé » démasqué par QA en mutant le seul `WHERE`). **La revendication #127/#143 vaut aussi pour le REÇU (corps de PR / message de commit), pas seulement le code/tests** : un reçu peut sur-revendiquer « effet observable runtime » (#127) pour un ⚙️ que la story ne fait que **déclarer** (consommation différée à une story ultérieure) alors même que le code+JSDoc sont honnêtes — décrire un ⚙️ seulement posé « config posée + validée, consommée en story X », **jamais** « AGIT ». **Corollaire feed-forward** : au découpage des AC, scoper la revendication « ⚙️ qui AGIT / test à effet observable runtime » à la story qui **CONSOMME** le param (fondation = déclare+valide ; consommateur = prouve l'effet mutation-prouvé), jamais à celle qui le pose — sinon l'AC fabrique un over-claim que le build hérite ; corriger via reformulation du corps de PR + message de squash-merge honnête, jamais amender un SHA vérifié+reviewé pour un libellé (rétro #155 : AC #148 « plafond ⚙️ qui agit » alors que la consommation budget est en 6.4 ; 4 reviewers ont convergé, QA CHANGES_REQUESTED levé par correction du seul libellé). **Quand l'orchestrateur ABSORBE un fix de consensus in-contract en commit ADDITIONNEL sur une PR existante (rétro #280, extension #164/#155 côté timing)** : mettre à jour le **body de la PR dans la MÊME action** (section « Fichiers touchés » + narration scope/Discovered) **AVANT** de (re)lancer les reviewers — le fix de code et sa doc dans le body sont **un acte atomique**, jamais séparés par un fan-out de review. Sinon le body raconte l'état **pré-absorption**, **contredit son propre diff** (le fichier absorbé reste marqué « hors scope / NON absorbé / absent de Fichiers touchés ») et **brûle un round de review complet** sur un body-mensonge #164 pourtant évitable (rétro #280 : 3/3 reviewers CHANGES_REQUESTED sur ce seul point, fond par ailleurs solide + approuvé ; #164 nomme déjà « corps de PR » comme reçu — le trou était le **timing**, pas la règle). **Worktree d'exécution de tests non partagé** avec un build qui l'édite en concurrence : une « flakiness » observée pendant des édits concurrents est un **artefact** — re-vérifier sur un arbre **quiescent** (build terminé) avant de la traiter comme réelle (rétro #126). **Un reviewer subagent ne SWITCHE JAMAIS l'arbre git partagé pour lire une PR** : lire **au ref** (`gh pr diff <n>`, `git show <ref>:<path>`, `git diff main...<branche>`) ou en **worktree isolé** (checkout détaché + `node_modules` symlinké RO) — un `git checkout <branche>` dans le repo où l'orchestrateur/d'autres reviewers travaillent en parallèle provoque un **artefact de working-tree** (fichier vu dans un état pré-edit alors que le commit est intact sur origin — rétro #147, symétrie lecture de #126) ; la vérité durable est le **commit poussé sur origin**, jamais le working-tree pendant un fan-out. **`gh pr diff <n>` sur une PR MULTI-COMMITS n'est PAS un diff net fiable (rétro #210, companion #147/#190)** : un fichier **ajouté puis retiré** sur deux commits peut y apparaître **net-ajouté** (artefact d'outil) → ne jamais conclure « le reçu ment » sur `gh pr diff` seul ; confirmer le diff net via **`git diff <base>...<head>` natif** (+ `git show <ref>:<path>` = *does not exist* pour un fichier retiré). En #210 QA a suspecté un mensonge de reçu plusieurs minutes avant que le diff natif ne confirme le reçu exact. **Verdict de reviewer = compté seulement s'il est ON-TASK** (cite la PR/story/specs + verdict argumenté) : un subagent de review qui renvoie un output générique/hors-sujet (brief non traité) = **review non faite** → relancer, ne jamais compter un silence ou un « approve » qui ne prouve pas la lecture de la story (symétrie avec la vérif du reçu, côté review — rétro #127 : game-design misfire au 1er lancement).

Un panel de tests/reviews vert sur un changement front **jamais rendu visible** = **DoD non satisfait** (#170 : 4/4 APPROVE sur un trait invisible car tous ont vérifié le token, aucun les pixels). **« Déclaré ≠ rendu/consommé » à l'échelle de l'ÉPIC (rétro #180, extension #125/#170)** : à la clôture d'un épic, vérifier que sa **VALEUR PRODUIT CENTRALE atteint réellement l'enfant** (effet observable bout-en-bout), pas seulement que chaque story a livré son mécanisme testé. Un épic entier peut passer toutes ses stories (100% coverage, mutation-prouvé, AC-littérales vertes) et rester **INVISIBLE au joueur** si le câblage consommateur n'est jamais fait (rétro #180 : épic #6 pipeline mondes complet — génération/buffer/QA/socle — mais `resolveWorld`/`--world-accent` jamais câblés au rendu carte → carte neutre, `PRODUCT.md` « thèmes IA = moteur de la boucle » non vécu). Ce gap est vu par les reviewers **FIDÉLITÉ** (game-design + PO), **jamais** par les reviewers ingénierie qui APPROUVENT le mécanisme (prolonge #95/#123 à l'échelle épic). Corollaire : une **story de câblage consommateur** qui rend l'épic visible se planifie **AVANT la clôture** (pas reléguée en backlog-hygiène), et tout **owner-gate se FILE en issue `needs-owner`** (pas seulement en corps de PR — sinon perdu au drain de clôture). **Faux point d'injection (rétro #184, extension #180 côté mécanisme d'injection)** : un **point de branchement documenté « fourni/branché à l'exécution owner »** (loader d'octets, `writeAsset`, secret, réf image…) doit être **RÉELLEMENT injectable** (dans les `deps`), jamais un **stub appelé en dur** — sinon c'est un **faux point d'injection** qui rend l'owner-run/consommateur **inexécutable sans changement de code**, et la story fondation croit à tort le mécanisme branchable (rétro #184 : `masterRefBytes` = stub honnête en JSDoc mais **appelé en dur dans `generateWorld`, hors `deps`** → gate owner #181 « générer les vrais mondes » **inexécutable fidèlement** — Teddy non ancré sur le master ADR 0009 → hors-style ; découvert en **instruisant** le gate, pas en review, avant dépense). Au découpage d'une **story fondation** posant un mécanisme « consommé/branché plus tard », auditer que **tous** ses points de branchement owner-run/consommateur sont **injectables** (pas seulement documentés comme tels), et que le **défaut committé** reste inerte/sûr (marqueur, no-I/O) pour 0 régression CI. Au découpage d'une story de reflow/modification, vérifier que l'écran/surface ciblé EXISTE en code (grep route/composant) avant de le scoper — nommé-en-spec ≠ construit (rétro #271, extension #125/#180).

Le feed-forward de brief doit alors trier la story-sœur par **surface de RISQUE** (ici : fidélité-modèle) et pas seulement par surface de code, et reformuler explicitement son piège-fidélité.
- **Décisions d'archi** : si ça touche une **spec contrat / data / dépendance / transverse** → **ADR** (`docs/adr/`) + design doc si `needs-design`. La review **signale**, l'**ADR décide** (hors PR). **Dans le contrat** (HOW dans le WHAT) = l'agent orchestrateur accepte en autonomie (mineurs+majeurs) ; **drift** (décisions verrouillées PLAN / pédagogie / éco / sécurité / scope) = **ton sign-off**. ADR accepté → **la spec est mise à jour** (canonique).
- **Skills = playbooks** (ne pas improviser) : **`orchestrate`** (démarrage de session, boucle autonome) → `story-start` → `open-pr` → `retro`, `discovered-issue` (triage hors-scope), `adr` (décisions contrat) ; **`next-dev-loop` obligatoire avant PR** (vérif runtime) ; `next-cache-components-*` pour la perf (différé) ; `frontend-design`/`design-tokens` pour l'UI ; `code-review`/`design-review`/`security-review` en review ; `brief-to-tasks` pour le découpage. **Exclure `expo:*` (natif) et `stripe:*` (pas de paiement).**
