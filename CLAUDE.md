# CLAUDE.md — multiplyz

## Projet
Jeu web de maths **ludique** pour aider une fille de **8 ans (CE1→CE2)** à combler ses lacunes : **compléments à 10, addition, soustraction, multiplication**. Jeu **sans fin**, collection de créatures, mascotte **Teddy** (son doudou Steiff). **App familiale perso** (pas un produit public).

## État
**Greenfield : specs complètes, code pas encore scaffoldé.** → **Lire la spec concernée avant d'implémenter une feature.**

## Stack (détail : `STACK.md`)
Next.js (App Router, **runtime Node**, pas edge) + React + TS + **Tailwind v4** + `tokens.css`. **SQLite local** (`better-sqlite3`, WAL) + **Drizzle**. Zustand. **PWA online-first**. Hébergement **VPS OVH via Laravel Forge** (daemon Node `next start` + Nginx + Let's Encrypt). Génération d'assets IA via **worker daemon**.

## Règles non négociables (en codant)
- **Tokens** : jamais de valeur en dur (couleur/espacement/typo/rayon) → `var(--…)` ou alias Tailwind. Source = `tokens.css`.
- **Copy** : français, **tutoiement**, posture croissance (« pas encore », jamais « faux »), voix de Teddy. **Strings centralisées** (zéro texte en dur).
- **Pédagogie** : **no-fail**, temps mesuré en silence, maîtrise = **juste + rapide**. La logique maîtrise/sélection vit **côté serveur** → suivre `ENGINE.md`, ne pas réinventer.
- **Serveur/données** : serveur = **source de vérité**, **online-first**, **runtime Node**, écritures **idempotentes**, progression **monotone**. SQLite **WAL**.
- **Migrations (Drizzle/SQLite)** : jamais `ADD col NOT NULL` sans default (crash sur table **peuplée** ; CI migre une base vide → ne l'attrape pas). Colonne dérivée **calculée par l'app** (non exprimable en SQL, ex. `nameKey` NFC+locale) = **nullable + backfill applicatif** dans `runMigrations` (idempotent `WHERE col IS NULL` + `db.transaction()` atomique + **anti-collision testée par source** — intra-batch ET valeur déjà persistée). **Jamais `.notNull()` de type sur une colonne physiquement nullable** (+ snapshot désync) = **drift snapshot↔SQL proscrit** (rétro #91/#105) : cohérence **schema.ts↔snapshot↔SQL↔base** (`db:generate` **no-op**) ou **table rebuild** ; garde observable `PRAGMA table_info` sur la nullabilité. Éditer une migration **livrée** seulement si **aucune base non-réparable ne l'a jouée** (skip drizzle = **timestamp** → migration éditée **inerte** sur base déjà migrée). Contrat data modifié → **ADR**.
- **Économie** : jamais d'argent réel, **ne bloque jamais l'apprentissage**, doublon → éclats, dépenses **online only**.
- **Sécurité** : PIN **hashé** (argon2/bcrypt), **rate-limit**, single-tenant famille. Jamais de PIN en clair / côté client.
- **Assets IA** : prompt de base **verrouillé** (ART), un monde ne pose que `--world-accent`, **Teddy depuis photos réelles**, QA kid-safe + fallback.
- **A11y** : feedback **doublé d'icône** (daltonisme), cibles ≥ 44 px, `prefers-reduced-motion`. Couleur de statut = token texte **constant** (`--color-on-warning`…), jamais un token qui s'inverse par thème (contraste). **Scaffolds visuels** : `--color-text-inverse` réservé à un fond **accent plein** — jamais posé sur un fond **neutre** (piège récidivant #94→#102, rétro #104) ; tout glyphe/chiffre/trait de scaffold visible exige **≥1 test de contraste WCAG à effet observable** (résolution `tokens.css`, `var()` résolus, ratio ≥4.5:1 — pas seulement le nom du token). Pour ces scaffolds, une garde visuelle doit asserter la **géométrie rendue** et/ou la **couleur résolue**, jamais seulement la forme d'un path SVG ou des `data-*` (100% coverage vacuous sur ces derniers, rétro #104). Le test de contraste s'applique **par glyphe/trait rendu distinct**, pas une fois par famille de tokens (deux glyphes différents — étoile pleine/vide, actif/inactif — sur un même scaffold = **deux tests**) ; le **fond de référence** = le fond DOM réellement empilé derrière ce glyphe (un frère empilé n'a pas le fond du médaillon voisin — rétro #125). **Quand une story TOUCHE un écran, auditer TOUS les glyphes rendus qu'il porte** pour la règle de contraste, pas seulement les nouveaux — un glyphe hérité d'une story antérieure reste **livré** par cette PR (rétro #126 : étoiles de #64 sur l'écran de résultats de 5.5, décoratives `--color-star` sur fond neutre ~1.2:1). **Token nommé du wireframe** : un token déclaré dans `tokens.css` **sans consommateur DOM** signale un **élément manquant**, pas un élément prêt — quand un wireframe **nomme** une métaphore structurante (« chemin », « piste », « fil »), poser une garde à effet observable que le DOM **rend l'élément nommé et consomme son token** (déclarer le token ne prouve pas le rendu ; piège invisible aux tests unitaires et à la capture, rétro #125). Cette règle « **déclaré ≠ rendu/consommé** » s'étend au-delà des tokens : (a) tout **param ⚙️ nommé par une spec** comme comportement calibrable (longueur du boss, seuil, palier) — le déclarer en config + le couvrir par un test de **présence** ne prouve pas qu'il **agit** ; exiger un test à effet observable qui **exerce le vrai chemin runtime** et rougit si le comportement retombe au défaut (rétro #127 : `bossQuestionCount` inerte, boss joué comme un niveau normal de 10 Q derrière un panel gates/coverage/mutation vert — récidive fidélité-modèle #95/#123, vue par game-design, pas par l'AC-littérale PO) ; (b) toute **valeur de layout structurante chiffrée** nommée par un wireframe (nombre de colonnes, gabarit) — `auto-fill`/`auto-fit` n'honore pas un compte fixe ; garder par un test non-vacuous du **compte rendu** (rétro #127 : « 3 colonnes » WIREFRAMES §8 rendues 1-2 sur téléphone). **Rendu + token consommé + contraste résolu vert ≠ PIXEL VISIBLE (rétro #170)** : un élément peut être présent au DOM, consommer son token, passer sa garde de contraste ≥X:1 résolue **et rester invisible à l'écran** — recouvert par un frère opaque (**occlusion / z-index**), `height:0`, hors cadre, clippé, mauvaise géométrie `top/height`. **jsdom ne fait AUCUN layout** → aucun test unitaire n'attrape l'occlusion (#170 : trait `--map-node-path-color` ≥3:1 mais peint DERRIÈRE le médaillon `zIndex:1` → invisible ; 4/4 reviewers ont validé le token, aucun les pixels ; corrigé en déplaçant le SVG dans la gouttière `top:var(--map-node-size)`). Donc pour tout élément **superposé/positionné** (SVG, overlay, badge, `absolute`), la garde de contraste/rendu doit être **doublée** (a) d'une **capture Playwright dont les PIXELS sont réellement analysés** (ouvrir l'image, regarder — pas seulement la générer) ET/OU (b) d'une **garde E2E sur la géométrie rendue réelle** (`boundingClientRect` : l'élément n'est pas recouvert, sommet ≥ bas du frère opaque). « Token vert » prouve la mécanique, **jamais** la visibilité. **La géométrie RAISONNÉE ne prouve pas la non-occlusion non plus (rétro #190, extension #170)** : pour tout élément superposé/positionné, une **marge calculée « sûre »** (« l'avatar s'étend de −40 à +8px, gap 32px → aucune occlusion ») peut être **empiriquement fausse** — en #190 l'avatar Teddy de 48px débordait **réellement de 6px** sur le médaillon du nœud **amont** (gouttière 32px), et **5 reviewers + l'orchestrateur + le pixel-look sur capture fixture** l'ont tous déclaré sûr ; seul un **assert `boundingClientRect` en vrai navigateur** l'a attrapé. Donc : (a) ne jamais substituer un raisonnement géométrique (même convergent entre reviewers) à un assert de layout réel ; (b) garder la non-occlusion contre **TOUS les frères empilables** — l'élément « sous » l'overlay ET les frères **adjacents/amont** dans la gouttière (pas seulement l'élément courant). **Audit binaire au ref** : utiliser **`git cat-file -p <blob>`**, jamais `git show <ref>:<fichier>` piped — le proxy rtk corrompt la sortie binaire (octets ≥0x80 → U+FFFD) → faux « fichier/PNG corrompu » (rétro #190, #126/#147-class). **Token CSS dérivé (variante #184 côté CSS, rétro #182)** : un token `color-mix`/`calc` **référençant une autre var** déclaré à `:root` **ne se re-dérive PAS** sous une **surcharge inline descendante** de la var source (la substitution `var()` d'un custom-property se résout au niveau de **déclaration**, pas d'usage) — ex. `--world-bg-tint: color-mix(…var(--world-accent)…)` à `:root` reste **neutre** même quand `<main>` surcharge `--world-accent` inline. « Dérivé automatiquement » est alors un **faux-dérivé dormant** (invisible aux tests unitaires — jsdom ne résout pas `color-mix`) : re-déclarer le token dérivé **au niveau de la surcharge** de sa var source, ou l'exercer par une garde à effet observable (valeur calculée en vrai navigateur). Un commentaire qui affirme la dérivation per-état non atteinte = tell #164 à corriger.
- **Paramètres `⚙️`** (notés dans les specs) = à **calibrer**, centralisés dans un fichier de config, jamais figés en dur sans raison.
- **Tests/CI** : seuils coverage **armés** (`thresholds` + `all:true`) — un `exclude` ne doit **jamais vider le scope** (100% vacuous = gate creux) ; exclude réservé au boilerplate framework non testable. Toute **garde / branche de repli** (anti-adjacence, anti-doublon, anti-collision, borne exacte) exige **≥1 test à effet observable** — un test qui **échoue si la garde est retirée/mutée**, pas seulement qui l'exécute ; le 100 % de coverage (même non-vacuous, `all:true`) ne suffit pas à lui seul (rétro #60/#61, PR #77). **Structure procédurale/dérivée** (composée à la volée) dont les **positions servent de clé persistée** (ex. `progress.level_index` vs géométrie de carte) ⇒ la **géométrie (nombre d'éléments + positions) doit être INVARIANTE à l'état runtime** (ex. la dette ne change pas le nombre de nœuds, seul un attribut non-clé comme le `type` peut varier) — garde = **test d'invariance de géométrie à effet observable** (même sortie structurelle quel que soit l'état dynamique ; casse si un élément est ajouté/retiré dynamiquement — rétro #123). **Test de rollback/atomicité** : la panne induite doit frapper **l'opération protégée par la transaction** (2ᵉ écriture), **jamais** un statement **en amont** (lecture/garde de pré-condition) qui court-circuiterait avant que la 1ʳᵉ écriture ne soit tentée — sinon le test reste **vert sans la transaction** (vacuous, 100 % coverage trompeur). `DROP TABLE` d'une table **lue en amont** = **anti-pattern** ; casser plutôt l'**écriture gardée** (colonne/contrainte manquante) pour que la panne survienne *après* la 1ʳᵉ écriture. Preuve exigée = retirer `db.transaction` **casse précisément ce test** (rétro #122). Cette exigence vise les transactions **multi-écritures** (≥2 écritures — faire échouer la 2ᵉ *après* la 1ʳᵉ pour observer le rollback). Une transaction sur **écriture unique** n'a **aucun état partiel à annuler** ⇒ **aucun test de rollback non-vacuous n'existe** (il passerait avec **ET** sans transaction) ⇒ **documenter honnêtement la garde** (ne jamais sur-revendiquer un anti-TOCTOU/atomicité *testé* qu'aucun test ne prouve), **PAS** ajouter un test qui resterait vert sans la transaction (lui-même vacuous, donc interdit — rétro #124). Ce principe s'étend à **toute branche/garde sans effet observable distinct de son absence** (pas seulement les transactions à écriture unique) : un prédicat/condition **redondant avec la sémantique du moteur** (ex. `WHERE col IS NOT NULL` sur un index UNIQUE composite, redondant car SQLite traite déjà les NULL comme distincts) **n'est pas une garde testable** — le **SUPPRIMER (simplifier)**, jamais garder la branche en revendiquant une couverture ni ajouter un test qui resterait vert sans elle. « **Correct** » (sémantiquement juste) ≠ « **testable** » ≠ « **nécessaire** » (rétro #143). **Tell commentaire↔code (récurrent #162→#164)** : tout **commentaire / titre de test / corps de PR / message de commit** qui affirme un **mécanisme précis** (nom d'API, garde, seuil, invariant — ex. « idempotence via `onConflictDoNothing` » alors que le code fait `onConflictDoUpdate`) se vérifie en review : le code annoté doit le réaliser **littéralement**. Écart = **defect tell** à corriger **même sans bug fonctionnel** derrière (drift intention↔implémentation = piège de maintenance) — corriger le libellé (ou le code s'il est faux), jamais laisser l'affirmation mentir ; vaut pour le **reçu** autant que le code (prolonge #127/#143/#155 côté documentation). **Compte agrégé de « N tests rouges » dans un reçu = tell fragile (rétro #173, extension #143/#155)** : revendiquer une mutation-preuve par un **cardinal** (« retrait de la règle X → 6 tests rouges ») est un tell qui casse au moindre test ajouté partageant le seam muté (un nouveau test consommant le même `failInspector`/fixture fait passer 6→7 sans que le code soit faux — récidive 6.5 : reçu 9→3 puis 6→7). Revendiquer la mutation-preuve par **test(s) NOMMÉ(s) qui rougit(ssent)** (« le test `budgets DISTINCTS` rougit si les compteurs re-fusionnent »), jamais par un total agrégé — sinon l'exactitude #143/#155 impose un recompte manuel à chaque test ajouté sur le même seam (coûteux + oublié). Vaut pour le reçu du build ET les titres/commentaires de tests. **Reçu de mutation-preuve nommant l'assertion/ligne où le test rougit = vérifié en exécution PLEIN-FICHIER, pas isolé (rétro #186, extension #164/#173)** : un `spy.mockRestore()` **non gardé** (sans `try/finally`) posé après une assertion qui peut lever **fuit le spy dans le test suivant** et **DÉPLACE l'échec observé** (le downstream rougit ailleurs — ex. `undefined`/`toHaveLength` — pas à l'assertion revendiquée), rendant le reçu **littéralement faux** même si la régression reste attrapée (pas de faux-négatif CI, mais tell #164 + valeur de debug dégradée). Garder chaque `spyOn` en `try { … expect … } finally { spy.mockRestore() }` (ou `restoreMocks:true` global) pour que l'échec de mutation atterrisse à l'**assertion nommée**. **Chemin format-RÉEL non prouvé par fixtures null/placeholder (rétro #189, extension #180/#184)** : un chemin de rendu/câblage prouvé **uniquement** avec des fixtures `null`/placeholder laisse le **chemin format-réel non prouvé** — il peut rester **dormant** sans qu'aucun test ne rougisse, notamment si un **outil owner-run** (gitignoré, non committé) qui **écrit des données consommées par le code committé diverge du contrat de format** de ce code (divergence **invisible à la CI par construction** : données gitignorées + code committé correct + fixtures placeholder ; défaut #189 = `gen-socle-assets.local.ts` stockait le ref **absolu** `/generated/socle/…` alors que `isRenderableAssetRef` exige le **relatif** `socle/…` → `background:null` → chemin réel dormant, trouvé en **instruisant** le déblocage, pas en review). Exiger (1) un **test d'intégration au format RÉEL valide → sortie non-null** (pas seulement le cas null/placeholder — ET le cas format-fautif → null), et (2) que tout **outil owner-run producteur de données** honore **littéralement** le contrat de format du consommateur committé (mêmes refs/clés que le **défaut committé**, ici `defaultSocleWriteAsset`). **`pnpm format`** relancé après tout **rebase/résolution de conflit** avant push.

## Workflow de dev (OBLIGATOIRE — cf. `WORKFLOW.md`)
- **Démarrage de session = boucle autonome** (« continue multiplyz ») → lancer le skill **`orchestrate`** (cf. `docs/adr/0004-orchestration-autonome-planning.md`). L'agent est **autonome sur le planning** : synchro état → **triage du backlog** (bloquant / prochain-épic / backlog / gate-déploiement) → **découpage de l'épic** en stories → **choix + parallélisation** (surfaces disjointes en //, partagées séquencées) → boucle story-start→PR→reviews→merge→rétro, sans solliciter le proprio. **Escalade UNIQUEMENT sur le drift** (décisions verrouillées PLAN/pédagogie/éco/sécurité/scope). Rapporter les décisions de planning, ne pas attendre d'aval hors drift.
  - **Contexte & quota** (autonomie longue, cf. `orchestrate` §7) : **build délégué à un subagent isolé** par story (thread principal = conclusions only → contexte plat) ; **tiering modèle** (Opus orchestration/cœur · Sonnet reviewers/mécanique · Haiku exploration) ; **jamais couper au milieu d'une story** (frontière propre = après merge+rétro + checkpoint statut) ; **quota bas → pause propre** à la frontière + **reprise auto-programmée** (skill `schedule` : run unique à l'heure de reset → chaîne auto ; en headless, drift → issue `needs-owner`, pas de reprise). **Reçu du subagent de build = VÉRIFIÉ, jamais cru sur parole** (SHA poussé / gates verts / capture) : avant de créditer ou merger, `git log` doit montrer le commit **sur la branche poussée** + `gh pr view --json headRefOid` concorde + **re-run des gates soi-même** ; reçu **invérifiable** (SHA inexistant, HEAD inchangé) = **tâche non faite** (rétro #126). **SHA véridique ≠ revendications de test véridiques** : une revendication « mutation-prouvé / garde à effet observable » pour une garde précise doit être re-vérifiée en **mutant CETTE garde spécifiquement** (le `WHERE`, la 2ᵉ écriture, le token remappé — pas la feature entière) et en observant le rouge, avant d'être créditée ; ne revendiquer « mutation-prouvé » qu'après avoir muté la garde nommée, sinon écrire « testé (exécuté), mutation non vérifiée » (rétro #143 : un prédicat partiel « mutation-prouvé » démasqué par QA en mutant le seul `WHERE`). **La revendication #127/#143 vaut aussi pour le REÇU (corps de PR / message de commit), pas seulement le code/tests** : un reçu peut sur-revendiquer « effet observable runtime » (#127) pour un ⚙️ que la story ne fait que **déclarer** (consommation différée à une story ultérieure) alors même que le code+JSDoc sont honnêtes — décrire un ⚙️ seulement posé « config posée + validée, consommée en story X », **jamais** « AGIT ». **Corollaire feed-forward** : au découpage des AC, scoper la revendication « ⚙️ qui AGIT / test à effet observable runtime » à la story qui **CONSOMME** le param (fondation = déclare+valide ; consommateur = prouve l'effet mutation-prouvé), jamais à celle qui le pose — sinon l'AC fabrique un over-claim que le build hérite ; corriger via reformulation du corps de PR + message de squash-merge honnête, jamais amender un SHA vérifié+reviewé pour un libellé (rétro #155 : AC #148 « plafond ⚙️ qui agit » alors que la consommation budget est en 6.4 ; 4 reviewers ont convergé, QA CHANGES_REQUESTED levé par correction du seul libellé). **Worktree d'exécution de tests non partagé** avec un build qui l'édite en concurrence : une « flakiness » observée pendant des édits concurrents est un **artefact** — re-vérifier sur un arbre **quiescent** (build terminé) avant de la traiter comme réelle (rétro #126). **Un reviewer subagent ne SWITCHE JAMAIS l'arbre git partagé pour lire une PR** : lire **au ref** (`gh pr diff <n>`, `git show <ref>:<path>`, `git diff main...<branche>`) ou en **worktree isolé** (checkout détaché + `node_modules` symlinké RO) — un `git checkout <branche>` dans le repo où l'orchestrateur/d'autres reviewers travaillent en parallèle provoque un **artefact de working-tree** (fichier vu dans un état pré-edit alors que le commit est intact sur origin — rétro #147, symétrie lecture de #126) ; la vérité durable est le **commit poussé sur origin**, jamais le working-tree pendant un fan-out. **Verdict de reviewer = compté seulement s'il est ON-TASK** (cite la PR/story/specs + verdict argumenté) : un subagent de review qui renvoie un output générique/hors-sujet (brief non traité) = **review non faite** → relancer, ne jamais compter un silence ou un « approve » qui ne prouve pas la lecture de la story (symétrie avec la vérif du reçu, côté review — rétro #127 : game-design misfire au 1er lancement).
- **Epic → Story** sur **GitHub** (Issues + Project). Chaque story = **critères d'acceptation** + DoD.
- Dev en **git worktree** isolé. **Toute tâche se termine par une PR** documentée + **captures Playwright**.
- **DoD** : lint + tests (**100 % logique critique**, pragmatique UI) + type-check/build + critères d'acceptation + **reviews agents approuvées** (Backend · Frontend+A11y · Security · QA/Test · Game-design · **Product Owner**). **Front livré ⇒ VÉRIF VISUELLE des pixels OBLIGATOIRE (garde-fou dur, rétro #170)** : générer la capture Playwright **ne suffit pas** — l'orchestrateur ET le reviewer Frontend **ouvrent la capture et regardent** que le changement est réellement **visible** (bon endroit, non recouvert), et un élément superposé exige en plus une **garde E2E de géométrie rendue** (non-occlusion). Un panel de tests/reviews vert sur un changement front **jamais rendu visible** = **DoD non satisfait** (#170 : 4/4 APPROVE sur un trait invisible car tous ont vérifié le token, aucun les pixels). **« Déclaré ≠ rendu/consommé » à l'échelle de l'ÉPIC (rétro #180, extension #125/#170)** : à la clôture d'un épic, vérifier que sa **VALEUR PRODUIT CENTRALE atteint réellement l'enfant** (effet observable bout-en-bout), pas seulement que chaque story a livré son mécanisme testé. Un épic entier peut passer toutes ses stories (100% coverage, mutation-prouvé, AC-littérales vertes) et rester **INVISIBLE au joueur** si le câblage consommateur n'est jamais fait (rétro #180 : épic #6 pipeline mondes complet — génération/buffer/QA/socle — mais `resolveWorld`/`--world-accent` jamais câblés au rendu carte → carte neutre, `PRODUCT.md` « thèmes IA = moteur de la boucle » non vécu). Ce gap est vu par les reviewers **FIDÉLITÉ** (game-design + PO), **jamais** par les reviewers ingénierie qui APPROUVENT le mécanisme (prolonge #95/#123 à l'échelle épic). Corollaire : une **story de câblage consommateur** qui rend l'épic visible se planifie **AVANT la clôture** (pas reléguée en backlog-hygiène), et tout **owner-gate se FILE en issue `needs-owner`** (pas seulement en corps de PR — sinon perdu au drain de clôture). **Faux point d'injection (rétro #184, extension #180 côté mécanisme d'injection)** : un **point de branchement documenté « fourni/branché à l'exécution owner »** (loader d'octets, `writeAsset`, secret, réf image…) doit être **RÉELLEMENT injectable** (dans les `deps`), jamais un **stub appelé en dur** — sinon c'est un **faux point d'injection** qui rend l'owner-run/consommateur **inexécutable sans changement de code**, et la story fondation croit à tort le mécanisme branchable (rétro #184 : `masterRefBytes` = stub honnête en JSDoc mais **appelé en dur dans `generateWorld`, hors `deps`** → gate owner #181 « générer les vrais mondes » **inexécutable fidèlement** — Teddy non ancré sur le master ADR 0009 → hors-style ; découvert en **instruisant** le gate, pas en review, avant dépense). Au découpage d'une **story fondation** posant un mécanisme « consommé/branché plus tard », auditer que **tous** ses points de branchement owner-run/consommateur sont **injectables** (pas seulement documentés comme tels), et que le **défaut committé** reste inerte/sûr (marqueur, no-I/O) pour 0 régression CI.
- **Boucle de review** jusqu'au DoD, puis **l'agent orchestrateur merge** (reviews scope+PO ✅ + CI verte + branche à jour) ; le proprio = **arbitre du drift**, peut révoquer/reprendre la main (cf. `docs/adr/0003-agent-merge-et-accepte-adr.md` [merge/ADR] + `0004-orchestration-autonome-planning.md` [planning/triage/séquencement]).
- **Gates en dur, ne jamais contourner** : GitHub Actions required checks + branch protection + CODEOWNERS + hooks `settings.json` (pre-PR + scope-guard).
- **Anti-drift** : les specs sont le **contrat** ; tout écart = rejeté en review. Rester dans le scope de la story.
- **Avant de coder** : lire **`LEARNINGS.md`** (leçons accumulées) + les specs du scope. **Découverte hors scope** → issue `discovered` (jamais absorbée). **Fin de story** → rétro → `LEARNINGS.md`. **Feed-forward de brief** (confirmé rétro #96, PR #112) : au découpage/briefing d'une story, identifier la story-sœur la plus récente à **surface comparable** et injecter explicitement sa **cause-racine de blocage/re-review** comme contrainte du brief — un simple renvoi générique à LEARNINGS ne suffit pas à éviter la récidive, la reformulation explicite dans le brief si. **Story à modèle pédagogique** (ENGINE/MAP/étayage) : la **fidélité au modèle** est une dimension de review **distincte de l'ingénierie** — game-design review le **MODÈLE** (le comportement est-il fidèle à la spec pédagogique ?), pas seulement l'AC-littérale ni les gates. **Un 100 % coverage + mutation testing ne prouve QUE la mécanique, jamais la fidélité au modèle** (les tests peuvent encoder fidèlement le mauvais comportement — rétro #95/#123). Le feed-forward de brief doit alors trier la story-sœur par **surface de RISQUE** (ici : fidélité-modèle) et pas seulement par surface de code, et reformuler explicitement son piège-fidélité.
- **Décisions d'archi** : si ça touche une **spec contrat / data / dépendance / transverse** → **ADR** (`docs/adr/`) + design doc si `needs-design`. La review **signale**, l'**ADR décide** (hors PR). **Dans le contrat** (HOW dans le WHAT) = l'agent orchestrateur accepte en autonomie (mineurs+majeurs) ; **drift** (décisions verrouillées PLAN / pédagogie / éco / sécurité / scope) = **ton sign-off**. ADR accepté → **la spec est mise à jour** (canonique).
- **Skills = playbooks** (ne pas improviser) : **`orchestrate`** (démarrage de session, boucle autonome) → `story-start` → `open-pr` → `retro`, `discovered-issue` (triage hors-scope), `adr` (décisions contrat) ; **`next-dev-loop` obligatoire avant PR** (vérif runtime) ; `next-cache-components-*` pour la perf (différé) ; `frontend-design`/`design-tokens` pour l'UI ; `code-review`/`design-review`/`security-review` en review ; `brief-to-tasks` pour le découpage. **Exclure `expo:*` (natif) et `stripe:*` (pas de paiement).**
