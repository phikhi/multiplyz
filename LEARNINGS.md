## Rétro story #29 — épic #2 Auth-lite (2.1 fondation, PR #34)

### 2026-07-01 — [db/coverage] Le callback table-extras de drizzle casse le gate 100 % fonctions (PR #34)
- Problème : le 3ᵉ argument de `sqliteTable(name, cols, (t) => [index/uniqueIndex/check(...)])` n'est **jamais invoqué au runtime** (seul drizzle-kit l'appelle au `generate`) → v8 le compte comme fonction **non couverte** → gate `functions: 100` rouge, `all:true`. `getTableConfig(t).indexes` **ne le couvre pas**. À l'inverse, le callback FK `.references(() => other.id)` **est** couvrable en forçant `getTableConfig(t).foreignKeys[0].reference()` dans un test.
- Leçon : sous gate 100 % fonctions, **éviter le callback extras** (index/check) dans le schéma. Faire respecter ces contraintes **au niveau requête** (ex. unicité prénom insensible à la casse via `lower(name)` en #2.2/#2.3) ou, si l'index DB est vraiment requis, assumer un seuil par-fichier documenté. Un index secondaire sur une table minuscule single-tenant = premature → l'omettre.
- Action : rappel (pattern à réappliquer dès #30 qui touche le schéma). Test FK couvert via `getTableConfig(...).foreignKeys[].reference()`.

### 2026-07-01 — [auth/deps] argon2id : `@node-rs/argon2` + piège `isolatedModules` (PR #34)
- Problème : `argon2` (npm ranisalt) = node-gyp → friction native pnpm 10 (cf. leçon #19). `@node-rs/argon2` = binaires **prebuilt** (napi) → pas de compilation. MAIS importer son enum `Algorithm.Argon2id` casse le build : `TS2748 Cannot access ambient const enums when 'isolatedModules' is enabled`.
- Leçon : préférer `@node-rs/argon2` (prebuilt) + l'ajouter à `serverExternalPackages`. Ne **pas** importer son `const enum` ; s'appuyer sur sa **variante par défaut (argon2id)** et **asserter le préfixe `$argon2id$`** en test.
- Action : rappel (référence pour toute dép exposant un `const enum` sous isolatedModules).

### 2026-07-01 — [config/next] `server-only` incompatible avec les modules partagés hors-Next (PR #34)
- Problème : review sécurité a suggéré `import "server-only"` sur `server-config.ts` (défense en profondeur). Mais ce module est **volontairement** consommé hors runtime Next (script `tsx db:migrate`, drizzle-kit, vitest) où `server-only` **throw** (paquet même pas installé + pas de condition `react-server`) → casserait migration + tests.
- Leçon : `server-only` ne peut coiffer que des modules importés **uniquement** dans le runtime Next. Un module de config partagé avec l'outillage hors-Next ne peut pas le porter. La protection Next (env non `NEXT_PUBLIC_*` jamais bundlé client) reste le garde-fou effectif.
- Action : rappel (finding review décliné avec raison).

### 2026-07-01 — [coverage] Aléa non biaisé = couverture déterministe (PR #34)
- Problème : générer un code via rejection-sampling (`if (byte < limit)` sur `randomBytes`) introduit une **branche probabiliste** — la branche « octet rejeté » peut ne pas s'exécuter dans un run → gate `branches: 100` **flaky**.
- Leçon : utiliser `crypto.randomInt(n)` (CSPRNG **non biaisé**, sans branche) plutôt qu'un modulo/rejet manuel → pas de branche à couvrir, déterministe, et pas de biais de modulo (bonus sécu).
- Action : rappel.

### 2026-07-01 — [process] Boucle de review propre en salve parallèle (PR #34)

## Rétro story #30 — épic #2 Auth-lite (2.2 onboarding, PR #36)

### 2026-07-01 — [db/concurrence] TOCTOU : check-then-write avec un `await` entre = transaction synchrone (PR #36, MAJOR backend)
- Problème : `createHousehold` faisait `if (householdExists) …` PUIS `await Promise.all([hash argon2])` PUIS `insert`. Le hash async **rend la main à l'event loop** → deux soumissions concurrentes de la server action (endpoint POST public ; le `disabled={submitting}` client n'est PAS une garantie serveur) passent toutes deux la garde → **2 propriétaires** (viole l'invariant « owner unique ») ou violation `UNIQUE(name)` re-levée en 500. Le code se réclamait « idempotent » sans l'être sous concurrence.
- Leçon : sous un invariant d'unicité/idempotence, tout **check-then-write avec une opération async intercalée** doit re-vérifier + écrire dans une **transaction SYNCHRONE better-sqlite3** (`db.transaction((tx) => {…})`, callback sans `await`) — hacher AVANT. Le callback sync s'exécute d'un bloc (BEGIN…COMMIT) sans reprise d'event loop → sérialisation. Bonus couverture : garder la vérif **uniquement** dans la transaction (pas de court-circuit externe redondant) → les 3 branches (return false / throw / insert) sont couvrables déterministe.
- Action : **candidat promotion CLAUDE.md** (règle serveur « écritures idempotentes » → préciser « check-then-write avec await = transaction sync »). À réappliquer dès #2.3 (création session/login) et toute écriture monotone.

### 2026-07-01 — [coverage] L'optional-chaining `?.` compte comme une branche v8 (PR #36)
- Problème : gérer le focus a11y via `useEffect(() => headingRef.current?.focus(), [step])` laisse la branche « current === null » **non couverte** (le titre est toujours monté) → gate `branches: 100` rouge.
- Leçon : préférer un **ref-callback** (`ref={useCallback((node) => node?.focus(), [])}`) : au changement d'étape l'ancien titre démonte (`node=null`) et le nouveau monte (`node=élément`) → **les 2 branches du `?.` sont exercées** naturellement, et le focus suit l'étape. Éviter `?.` / paramètres par défaut / ternaires dont un côté est inatteignable sous gate 100 %.
- Action : rappel (réapplique la discipline « pas de branche morte » déjà notée #34).

### 2026-07-01 — [a11y] Assistant multi-étapes : focus au montage + région live pour secret à usage unique (PR #36, MAJOR frontend)
- Problème : au changement d'étape, le focus retombait sur `<body>` (nouveau `<h1>` jamais annoncé) ; le **code de secours à usage unique** était un simple `<p>` (ni focusable ni annoncé) → un utilisateur clavier/lecteur d'écran pouvait le **rater**.
- Leçon : dans un wizard, déplacer le focus sur le titre de chaque étape (`<h1 tabIndex={-1}>` + ref-callback focus au montage) ; annoncer tout contenu critique éphémère via `role="status"`. Pattern réutilisé par la connexion #2.3 (PinPad).
- Action : rappel a11y (à réappliquer #2.3).

### 2026-07-01 — [e2e] Gating dépendant d'un état DB → base E2E dédiée wipée à froid + spec mutante `retries:0` (PR #36)
- Problème : la racine `/` affiche l'onboarding **seulement si aucun foyer** ; les E2E doivent partir d'un état vide déterministe, et le test qui **crée** le foyer mute le single-tenant partagé. Un retry CI (`retries:1`) sur ce test rejouerait sur un foyer déjà créé → `alreadyConfigured` → jamais l'écran attendu.
- Leçon : base SQLite **dédiée E2E** (`data/e2e.sqlite`, jamais la dev), wipée en `globalSetup` + **migrée au boot** (`db:migrate && dev`) ; `describe.serial` + `test.describe.configure({ retries: 0 })` pour le spec mutant (un retry ne récupère pas une écriture one-time) ; donner un `<h1>` au placeholder « foyer configuré » pour que les specs cold-start `/` restent vertes dans les 2 états.
- Action : rappel (référence pour toute story dont un écran dépend d'un état serveur).

### 2026-07-01 — [security/copy] Garde de forme sur server action publique + registre parent neutre (PR #36)
- Problème : (a) `"use server"` = endpoint public ; `input` typé n'est pas garanti au runtime → `sanitizeName(input.name)` sur un `name` non-string = `TypeError` 500 au lieu d'une erreur de validation (AUTH §4). (b) Le tutoiement de Teddy avait fuité dans l'étape **parent** (`parentPin.method`, `errors.PARENT_PIN_SAME`), alors que COPY §5 impose un registre **neutre** côté parent.
- Leçon : (a) garder la **forme** (types des champs) en tête de toute server action avant sanitisation → erreur de validation propre. (b) surveiller le **registre par audience** (Teddy/tutoiement enfant vs neutre parent) — un même écran onboarding mélange les deux.
- Action : rappel (réapplique à #2.3+ et à tout copy parent/enfant).

### 2026-07-01 — [dx] 3 footguns de dev local révélés dès que `/` lit la DB (PR #40, #41)
- Problème : dès que la racine `/` lit une table à la requête (gating #30), le dev local casse là où les stories précédentes (page statique) ne touchaient pas la DB : (a) `pnpm dev` ne migrait pas → `SqliteError: no such table: profiles` (base dev fraîche) ; (b) le **service worker** cachait `/_next/static/*` en **cache-first** (« assets immuables ») → **faux en dev** : les chunks Turbopack réutilisent leurs URLs après édition → un chunk **périmé** servi → `strings.onboarding undefined` / TypeError ; (c) `prettier --check <un-seul-fichier>` a donné un **faux vert** alors que `prettier --check .` (la commande CI) échouait sur le même fichier → `quality` rouge au push.
- Leçon : (a) `pnpm dev` = `pnpm db:migrate && next dev` (idempotent ; prod migre via Forge, pas au boot) ; (b) SW `/_next/static` en **réseau-d'abord** (frais en ligne dev+prod, cache en repli hors-ligne), bumper `CACHE_NAME` pour purger l'ancien cache ; cache-first n'est correct QUE sur des noms hashés (prod) ; (c) toujours lancer **`pnpm format`** (commande CI exacte) avant push, pas un check par-fichier.
- Action : (a)+(b) appliqués. (c) rappel — candidat **hook pre-PR** `pnpm format` (déjà pressenti epic #1).

## Rétro story #32 — épic #2 Auth-lite (2.4 rate-limit + backoff, PR #46)

### 2026-07-01 — [schema/coverage] Clé composite en PK texte encodée → évite le callback d'extras drizzle (PR #46)
- Problème : une table de compteurs (rate-limit par profil/IP) veut une clé composite `(scope, key)`. Un **PK composite** ou un **UNIQUE index** en drizzle passe par le 3ᵉ argument callback de `sqliteTable` → jamais invoqué au runtime → casse le gate 100 % fonctions (LEARNINGS #34).
- Leçon : encoder la clé composite dans **une seule colonne PK texte** (`"<scope>:<clé>"`, ex. `"profile:5"` / `"ip:1.2.3.4"`), l'assemblage se faisant dans une fonction pure testée (`attemptKey`). Aucun callback d'extras → schéma couvrable à 100 %, et l'upsert `onConflictDoUpdate` cible ce PK simple.
- Action : rappel (pattern pour toute table clé-valeur / compteur / jonction sous gate 100 % fonctions).

### 2026-07-01 — [security/deploy] Rate-limit par IP : X-Real-IP de confiance, pas le 1er maillon de XFF (PR #46, finding backend majeur)
- Problème : lire le **1er maillon** de `X-Forwarded-For` suppose que Nginx **écrase** l'en-tête. Or le template Nginx/Forge par défaut **ajoute** (`$proxy_add_x_forwarded_for`) le `remote_addr` réel APRÈS la valeur cliente → le 1er élément est **contrôlable par le client** → rate-limit par IP contournable (fausse IP/req) ou empoisonnable (bloquer une IP tierce). Le garde **par profil** reste efficace (défense primaire).

Fix : arc SVG quadratique (Bézier) tracé du tick départ au tick arrivée + pointe de flèche orientée par l'arithmétique (`forward = correctAnswer > a`), même référentiel de coordonnées que les graduations (fraction `(v−min)/(max−min)`), décoratif `aria-hidden` (contrat #93/#94 intact).
- Leçon : (1) pour un **étayage visuel pédagogique**, la review doit vérifier que **le geste visuel cœur du modèle** (ce à quoi sert l'étayage) est **réellement rendu**, pas seulement que les données/labels sont corrects. L'AC-littérale (« affiche add/sub avec la bonne direction ») peut passer alors que la **fidélité au modèle** (« l'enfant VOIT le saut ») échoue. (2) La **lentille game-design est décisive sur la fidélité au modèle** même 1-contre-3 (verify≠vote, #91) : PO valide l'AC-littérale, game-design valide que la représentation *enseigne* vraiment. (3) La review d'un étayage visuel **exige de regarder les captures** (game-design a vu le défaut à l'œil ; le code seul « fonctionne »). L'orchestrateur regardant lui-même la capture régénérée = vérification bon marché avant merge.
- Action : **feed-forward brief #96 (matrice `mult`)** — le **regroupement** (`a` paquets de `b`) doit être **visuellement évident** (groupement spatial / bordures / lignes-colonnes distinctes), PAS juste `a×b` points alignés ; un enfant doit VOIR « 6 paquets de 8 ». Exiger une **garde observable du geste visuel** (le groupement existe et dérive des operands) + **review sur captures**. Généraliser : tout étayage visuel = « le modèle est-il DESSINÉ, pas seulement décrit ? ».

### 2026-07-03 — [frontend] Aligner un overlay SVG sur une rangée d'éléments positionnés en flex : partager le MÊME référentiel de fraction `(v−min)/(max−min)` + `viewBox="0 0 100 100"`/`preserveAspectRatio="none"` (PR #102)
- Observation : le connecteur de saut (arc SVG) devait s'aligner **exactement** sur des graduations positionnées en `left: frac%` dans une rangée flex `overflow-x:auto`. Solution propre : l'arc et les ticks calculent leur position via la **même** fonction `fractionOf(v) = (v−min)/(max−min)` ; le `<svg viewBox="0 0 100 100" preserveAspectRatio="none">` mappe `x = frac×100` → alignement pixel-exact sans casser le reflow responsive déjà validé (l'overlay ne perturbe pas la layout flex). Contrainte `LINE_MARGIN≥1` garantit `max≠min` (branche morte `max===min` retirée pour garder 100 % honnête, LEARNINGS #62).
- Leçon : pour superposer un tracé vectoriel sur des éléments flow/flex, dériver les deux du **même modèle de coordonnées normalisé** (fraction 0..1) et laisser le SVG s'étirer (`preserveAspectRatio="none"`) plutôt que de recalculer des pixels — l'overlay reste aligné et n'impacte pas la layout responsive.
- Action : rappel frontend — étayages à overlay vectoriel (#96 matrice si lignes/liens) = partager le référentiel de fraction, overlay non-perturbant.

## Rétro story #105 — Fix migration 0005 `name_key` sur table `profiles` peuplée (discovered, PR #106)

### 2026-07-03 — [db/migration] `ALTER TABLE … ADD col NOT NULL` sans default est REFUSÉ par SQLite sur une table déjà peuplée ; une colonne dérivée calculée par l'app (non exprimable en SQL) = colonne **nullable** + **backfill applicatif** dans `runMigrations` (PR #106)
- Problème : la migration 0005 (#37) faisait `ALTER TABLE profiles ADD name_key text NOT NULL` sans default → `SqliteError: Cannot add a NOT NULL column with default value NULL` sur toute base `profiles` **déjà peuplée** (crash `pnpm db:migrate` ; marchait seulement sur table vide → passé inaperçu en CI qui migre une base fraîche). La valeur `name_key = nameKey(name)` (NFC + `toLocaleLowerCase("fr-FR")`) n'est **pas calculable en SQL** — `lower()` SQLite est ASCII-only (ADR 0005) → un backfill `SET name_key = lower(name)` serait faux pour les capitales accentuées.
- Leçon : (1) une migration qui ajoute une colonne `NOT NULL` doit soit un default, soit — si la valeur dépend d'un calcul applicatif — être ajoutée **nullable** puis **backfillée côté app** (ici `backfillNameKeys` dans `runMigrations`, après `migrate()`). (2) Le backfill doit être **idempotent** (`WHERE col IS NULL` → no-op au rejeu et sur base fraîche), **transactionnel** (`db.transaction()` → atomique, pas de base à demi-migrée), et **anti-collision** (détecter AVANT écriture une clé dérivée qui collisionne, lever une erreur explicite plutôt que laisser l'index UNIQUE échouer en plein `UPDATE`). (3) Une migration qui ne s'applique QUE sur table vide est un piège : la CI (base fraîche) reste verte, seul un dev à base pré-existante crashe.
- Action : promue en règle dure CLAUDE.md (§Serveur/données). Feed-forward : toute future migration ajoutant une colonne dérivée app-calculée suit ce patron (nullable + backfill idempotent/transactionnel/anti-collision testé).

### 2026-07-03 — [db/drift] Garder `.notNull()` de TYPE sur une colonne physiquement NULLABLE (+ snapshot `notNull:true`) = exactement le drift snapshot↔SQL déjà proscrit (ADR 0005 / rétro #91) : cohérence PLEINE sur 4 plans, ou table rebuild (PR #106, Backend BLOQUANT)
- Problème : 1ère version du fix gardait `text("name_key").notNull()` dans `schema.ts` (+ snapshot `notNull:true`) tout en rendant la colonne physique nullable dans le SQL — « le `.notNull()` reste le contrat de type ». Backend a bloqué : c'est **le drift snapshot↔SQL** que ce codebase a durci (rétro #91, ADR 0005 rejette explicitement « migration SQL à la main + snapshot désynchronisé »). `db:generate` reste no-op (il ne compare que schema.ts↔snapshot, jamais la base) → le drift est **silencieux**, non détecté par le gate.
- Leçon : sous SQLite on ne peut PAS avoir à la fois le `.notNull()` de type Drizzle ET une colonne physiquement nullable sans mentir au snapshot. Deux options cohérentes : (a) **nullable partout** — `schema.ts` sans `.notNull()` ↔ snapshot `notNull:false` ↔ SQL `ADD col text` ↔ base — le non-null devient un **invariant applicatif** (validation + INSERT + backfill) ; (b) **table rebuild** (12-step) pour un vrai `NOT NULL` physique. Retenu ici : (a) (rebuild impossible car la clé n'est pas SQL-calculable). **Garde observable** : `PRAGMA table_info(profiles)` → `name_key.notnull === 0` (rougit si un futur agent repasse la colonne `NOT NULL`).
- Action : promue en règle dure CLAUDE.md. Le contrat data change → **ADR 0005 amendé** (Addendum #105, spec canonique).

### 2026-07-03 — [qa] Une garde anti-collision à deux sources (collision intra-batch ET collision avec une valeur déjà persistée) exige **un test par branche** — un seul test intra-batch laisse la moitié « clé déjà posée » non mutation-tight (PR #106, QA)
- Problème : `backfillNameKeys` détecte les collisions via une Map `claimedBy` alimentée par (1) les lignes déjà non-NULL en base + (2) les lignes du batch courant. Le test de collision initial n'exerçait que le cas **intra-batch** (2 lignes NULL) → neutraliser le pré-chargement depuis les lignes non-NULL (`WHERE name_key IS NOT NULL` → `WHERE 1=0`) laissait **tous les tests verts** (garde à moitié non gardée). Déterministe → hors exception LEARNINGS « seulement atteignable en concurrence ».
- Leçon : une garde qui agrège **plusieurs sources** de conflit a **plusieurs branches** ; la règle « ≥1 test à effet observable par garde » (CLAUDE.md) se lit **par branche/source**, pas par garde globale. 100 % coverage ne suffit pas (les deux sources exécutent la même ligne de `throw`). Il faut un test qui isole chaque source (ici : +1 test « collision avec une clé `name_key` déjà posée en base »).
- Action : rappel QA — pour une garde multi-source, un test mutation-tight par source. Vérification faite ici en mutant chaque source séparément.

### 2026-07-03 — [db/feed-forward] Retirer `.notNull()` rend la colonne **optionnelle dans le type d'INSERT Drizzle** → TS n'attrape plus un INSERT profil qui oublie `name_key` (PR #106, Backend feed-forward)
- Observation : `name_key` étant désormais nullable, le type d'INSERT `profiles` rend le champ optionnel. Aujourd'hui seul `createHousehold` (`household.ts:147`) insère un profil et fournit toujours `nameKey: nameKey(name)` — OK. Mais un futur chemin d'insertion (ex. story « ajouter un profil frère/sœur ») pourrait **omettre `name_key` sans erreur TS** → ligne NULL qui passe l'index UNIQUE (NULLs distincts) et reste invisible au lookup `nameTaken` (`WHERE name_key = ?` ne matche pas NULL). Coût inhérent et assumé de l'alternative cohérente (documenté ADR 0005 comme invariant applicatif).
- Leçon : quand une colonne passe à nullable pour cohérence snapshot, la sûreté « champ requis à l'INSERT » du type Drizzle est perdue → l'invariant se maintient **au chemin d'écriture**, pas au type. À rappeler explicitement dans tout brief touchant un nouvel INSERT de la table concernée.
- Action : **feed-forward** — tout futur INSERT `profiles` (story multi-profils frères/sœurs, import, etc.) DOIT fournir `name_key = nameKey(name)` ; ajouter une garde (ex. test onboarding/insert qui asserte `name_key` non-NULL) au moment où un 2e chemin d'insert apparaît.

### 2026-07-03 — [process] Éditer une migration DÉJÀ livrée est licite quand : pas de prod + base locale déjà marquée appliquée (le migrator drizzle applique par **timestamp** `folderMillis`, pas par re-hash → la migration éditée est **inerte** sur une base l'ayant déjà jouée) (PR #106)
- Observation : le fix a **édité 0005** plutôt que d'ajouter 0006. Justification : une base pré-0005 crashe *sur* 0005 → un 0006 ultérieur ne s'exécuterait jamais (le seul moyen de réparer une base pré-0005 est de corriger 0005). Le migrator drizzle better-sqlite3 décide d'appliquer par `Number(lastDbMigration.created_at) < migration.folderMillis` (timestamp du `_journal.json`), **jamais** par comparaison de hash → sur une base ayant déjà appliqué l'ancien 0005, la version éditée n'est **pas rejouée** (inerte, pas d'erreur). Pas de prod → aucune base tierce impactée.
- Leçon : éditer une migration livrée est sûr **ssi** (pas de prod OU aucune base tierce ne l'a jouée) — sinon les bases l'ayant déjà appliquée ne verront jamais la correction et divergeront. Le skip drizzle est **timestamp-based**, utile à connaître pour raisonner sur l'inertie d'une migration éditée.
- Action : rappel — avant d'éditer une migration livrée, vérifier qu'aucune base non-réparable ne l'a déjà appliquée ; sinon, migration corrective en avant (nouveau fichier).

### 2026-07-03 — [process] Boucle de review saine : 2 reviewers bloquants indépendants (Backend=drift snapshot, QA=collision) convergents avec les notes *informational* de Security/PO → 1 round de fix consolidé, re-review par les auteurs du bloquant (PR #106)

C'est la **2e récidive** de ce piège précis après #95/4.3 (étayage droite numérique : AC-littérale + 3 reviewers verts, seul game-design avait vu que le « saut » n'était pas dessiné). Le motif commun : une implémentation **techniquement irréprochable** peut satisfaire l'AC au pied de la lettre tout en trahissant l'intention du modèle pédagogique/carte — seule la lentille game-design (qui compare au MODÈLE, pas à l'AC) l'attrape.
- Fix appliqué (PR #130, commit `0f6af77`) : **géométrie invariante** — le monde a TOUJOURS `levelsPerWorld + 1` nœuds aux mêmes positions, quel que soit l'état de la dette ; **overlay de type** sur le nœud **courant** (1ᵉʳ non terminé, si ≠ boss) plutôt qu'insertion d'un nœud — remédiation immédiate, `level_index` stable. Nouveau test d'invariance de géométrie (garde data) + preuve-mutation refaite (overlay retiré / retour au splice / boss-guard neutralisé / borne `>`→`>=` : chacun casse un test précis). Re-review game-design → APPROVE, merge.
- Action : rappel doctrine (2e occurrence — voir promotion proposée ci-dessous) — pour toute story à **modèle pédagogique** (ENGINE/MAP/étayage), le brief ET la review doivent traiter la **fidélité au modèle** comme dimension **distincte** de l'ingénierie ; ne jamais déduire d'un panel Backend/QA/PO 3/3 APPROVE que le modèle est fidèle — game-design doit se prononcer spécifiquement là-dessus avant merge.

### 2026-07-04 — [data/procédural] Une structure procédurale/dérivée dont les POSITIONS servent de clé persistée exige une géométrie invariante à l'état dynamique — sinon la clé se désynchronise entre deux calculs (PR #130)
- Problème : la carte (`buildMap`) est recomposée à la volée (pure, pas stockée), mais ses positions (`level_index`) servent de **clé persistée** dans `progress` (MAP §4). L'implémentation initiale faisait varier le **nombre de nœuds** selon un état dynamique (la dette de révision) → deux appels à `buildMap` avec la même dette différente produisent un `level_index` différent pour le même niveau logique, désynchronisant la progression persistée.
- Leçon : quand une structure est composée dynamiquement mais que ses **positions/index servent de clé de persistance**, la **géométrie** (nombre d'éléments + positions) doit être **INVARIANTE à l'état dynamique** — seuls des attributs *non-clés* (ici le `type` d'un nœud) peuvent varier avec cet état. La garde requise n'est pas seulement « déterminisme » (même entrée ⇒ même sortie) mais un **test d'invariance de géométrie à effet observable** : même nombre de nœuds/mêmes positions/mêmes `level_index` quel que soit l'état dynamique (dette, progression) — un test qui casse si un élément est ajouté/retiré dynamiquement.
- Action : rappel data — pour toute future structure procédurale indexée par une clé persistée (carte, arbre de compétences, etc.), poser explicitement ce test d'invariance de géométrie dès la 1ʳᵉ implémentation, pas seulement le test de déterminisme.

### 2026-07-04 — [process/brief] Écart de feed-forward de brief : les leçons d'ingénierie de la story-sœur #122 ont bien été injectées, mais pas la dimension fidélité-modèle ni le piège #95/4.3 — le brief n'a pas explicité que game-design review le MODÈLE, pas l'AC (PR #130)
- Problème : le brief de 5.2 injectait correctement les leçons **d'ingénierie** récentes (gardes observables mutation-prouvées de 5.1/#122, ⚙️ centralisés, déterminisme). Il n'injectait **pas** la leçon de la story-sœur la plus proche en **surface de risque** — #95/4.3, où game-design avait bloqué seul malgré un panel 3/3 vert pour une raison structurellement identique (AC-littérale satisfaite, modèle trahi). CLAUDE.md exige déjà ce feed-forward (« story-sœur à surface comparable → injecter sa cause-racine explicitement, pas un renvoi générique à LEARNINGS ») — mais le découpage du brief avait classé #122 (data/fondation) comme la sœur pertinente, pas #95 (fidélité pédagogique), parce que la **surface de code** (carte vs étayage visuel) diffère alors que la **surface de risque** (fidélité au modèle) est la même.
- Leçon : identifier la story-sœur pertinente pour le feed-forward de brief ne doit pas se faire par **similarité de code/surface technique** (carte ≠ étayage) mais par **similarité de risque** — ici, « touche un modèle pédagogique/spatial verrouillé (ENGINE/MAP) où l'AC peut être satisfaite sans que le modèle le soit ». Pour ce type de story, le brief doit **expliciter** que game-design review la fidélité au MODÈLE (pas seulement l'AC), et reformuler le piège-fidélité de la sœur la plus récente sur ce risque — même si sa surface de code semble éloignée.
